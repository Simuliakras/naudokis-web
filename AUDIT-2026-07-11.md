# Naudokis mobile-bridge release audit — 2026-07-11

## 2026-07-12 implementation follow-up

This follow-up supersedes the implementation-status statements in the original findings below. The release is still **conditional** because several launch checks require production credentials, deployed hosts, signed apps, physical devices, and qualified legal approval.

### Completed in the release candidate

- Synthetic `e2e-test` inventory is excluded from browse, feed, and direct listing routes. Failed listing images now render a localized branded fallback without layout shift.
- Empty inventory, offline, timeout, and API/server failures have distinct messaging. Search/filter state is preserved and failed requests have an explicit retry action.
- Google Maps is retained. The iframe is not requested until the visitor selects **Load Google Maps**, the preference is remembered locally, the privacy notice discloses the integration, and the consent path has automated coverage.
- Five focused transaction-policy pages are independently reachable in both LT and EN: payments/fees, cancellations/refunds, deposits/damage/disputes, reservation/handover, and trust/safety/support. They are linked from the legal hub, footer, sibling navigation, and sitemap.
- `/go` redirect outcomes, app-open attempts, fallback display, retries, QR/smart handoffs, renter booking intent, and owner listing intent are measured. A signed, expiring journey-token endpoint is available for native continuation and intent events; the native implementation contract is documented in `docs/app-handoff-analytics.md`.
- Marketplace image dimensions were tightened, the 149 KB bitmap phone mockup was replaced with a code-native locale-neutral preview, the mobile pattern request was removed, font loading was consolidated, and Sentry is deferred entirely when no public DSN is configured.
- A stricter script CSP is now emitted in report-only mode while the hydration-compatible enforced policy remains in place. Production reports are accepted by the CSP reporting endpoint. Next.js experimental SRI was evaluated and removed after it emitted a missing stylesheet under Turbopack; a production test now checks every referenced stylesheet for a successful CSS response.
- Android association configuration requires two unique SHA-256 fingerprints and returns a deployment error instead of publishing an incomplete statement. OneLink requires HTTPS and fails safely. `npm run verify:app-links` validates the local values and can validate the deployed association endpoints.
- All source `DRAFT` markers were resolved, LT/EN legal data was regenerated from canonical Markdown, and English acquisition imagery is now locale-neutral. Qualified product/legal approval remains an organizational gate rather than something established by this code change.

### Final local evidence

- Optimized Next.js 16.2.7 production build: passed.
- ESLint and TypeScript: passed with no errors.
- Production-readiness Playwright suite: **10/10 passed**, including no-JS search, hard 404s, policy routes, synthetic-data filtering/image fallback, Google Maps explicit loading, strict report-only CSP, signed native events, and emitted stylesheet integrity.
- Final throttled cold-cache mobile Lighthouse: **86 performance / 100 accessibility / 96 best practices / 100 SEO**; FCP **1.09 s**, LCP **4.13 s**, TBT **50 ms**, CLS **0**. This improves the original 77 score and 5.68 s LCP, while unused-JavaScript savings fell from roughly 75 KiB to **24 KiB**. The production field target remains p75 LCP ≤2.5 s.

## Severity-ranked master findings register

### Pre-launch blockers

#### REL-001 — Deployment and public-host smoke test remain pending

- **Status:** expected pre-deployment work, not a candidate defect.
- **Affected:** all public routes, headers, well-known files, CDN caching, and environment configuration.
- **Risk:** local correctness does not prove canonical-host rewrites, cache behavior, secrets, association-file content types, or production API behavior.
- **Action:** deploy atomically to staging or the launch host, invalidate relevant CDN objects, and execute the route/status/header/asset/device smoke matrix before opening traffic.
- **Acceptance:** every route class returns its intended status and canonical URL; environment-dependent integrations work; the candidate hash and visible content match the reviewed build.
- **Complexity:** M–L, infrastructure. **Launch gate: yes.**

#### DL-001 — iOS Universal Links require public-host validation

- **Affected:** `/.well-known/apple-app-site-association`; iOS Safari and in-app browsers.
- **Problem:** the repository contains the extensionless association file, but it cannot be validated on the intended host until this candidate is deployed. The old site’s response is not evidence against the candidate.
- **Impact:** signed iOS builds cannot claim first-party deep links; listing/referral intent falls back to browser/store behavior and context can be lost.
- **Fix:** serve the file at the exact extensionless path with 200, `application/json`, no redirect, no authentication, and matching Team ID/bundle ID/path rules.
- **Acceptance:** Apple CDN association validation and a physical-device tap from Notes, Messages, Safari, and Mail open the installed app to the intended listing/referral; uninstall case lands in the correct store and resumes context after install.
- **Complexity:** M, CDN + iOS entitlement. **Release blocking: yes.**

#### DL-002 — Android App Links signing configuration is incomplete

- **Affected:** `/.well-known/assetlinks.json`; Android Chrome/webviews.
- **Problem:** current source and Play URL use `com.naudokis.naudokis`, but the local release environment contains only one certificate fingerprint although configuration documentation requires Play signing and upload fingerprints. The old site’s placeholder file is outside this candidate’s assessment.
- **Impact:** Android verification fails, links show a chooser or remain in the browser, and deferred continuation is unreliable.
- **Fix:** confirm the shipping package ID, publish all signing certificate SHA-256 fingerprints, remove placeholders, and align app manifest intent filters with the same hosts/paths.
- **Acceptance:** `adb shell pm get-app-links` reports verified; taps from Chrome, Gmail, Messages, and common social webviews open the correct signed build; installed/uninstalled flows preserve target and attribution.
- **Complexity:** M, Android + release engineering. **Release blocking: yes.**

#### DL-003 — End-to-end native continuation is not yet proven

- **Affected:** `/go`, `/invite`, `/listing/*`, booking targets, QR codes, OneLink; iOS/Android.
- **Problem:** the web contract now preserves a safe internal `appPath` and campaign fields, but this environment cannot prove that both signed apps consume every path or that AppsFlyer deferred deep linking restores it after first install.
- **Impact:** the primary conversion can appear successful while opening only the app home, losing the listing, or losing campaign/referral attribution.
- **Fix:** run the device matrix in the release plan with production OneLink, signed builds, clean-install states, private browsing, social webviews, interrupted store sessions, refresh, and back navigation.
- **Acceptance:** every matrix row records expected app screen, attribution payload, fallback, and retry behavior; no loop, duplicate tab, or lost target.
- **Complexity:** L, cross-team QA. **Release blocking: yes.**

### Critical and high

#### DATA-001 — Test listing and broken image contaminate marketplace data

- **Affected:** homepage offer grid, feed, listing pages; all viewports.
- **Problem:** the API dataset used by the release candidate returns an `e2e-test` image URL that responds 403. Next image optimization logs repeated upstream failures and Lighthouse records a console error.
- **Impact:** broken imagery damages trust, creates noisy monitoring, and lowers performance/reliability signals. A test listing in a public catalogue is unacceptable at launch.
- **Fix:** remove/isolate synthetic records from production-like data, purge related CDN keys, add a durable image placeholder on upstream failure, and monitor image-origin error rate.
- **Acceptance:** no `e2e-test` records in production APIs; all first-page images return 200; forced 403/404 image test renders a branded fallback without console noise or layout shift.
- **Complexity:** M, backend/data + frontend. **Release blocking: yes.**

#### TRUST-002 — Essential transaction policies are not independently discoverable

- **Affected:** `/teisine`, `/naudojimosi-salygos`, `/privatumo-politika`; trust evaluation.
- **Problem:** public navigation exposes Terms, Privacy, and deletion information, while payment timing, deposit release, cancellation, refund, damage evidence, dispute escalation, and support service levels are mainly summarized in marketing copy or incorporated into broad Terms.
- **Impact:** high-intent renters and owners cannot independently verify the rules before installing; support and chargeback risk rises.
- **Fix:** legal/product review first, then publish stable, linkable policy sections or pages for payments/payouts, deposits/damage, cancellations/refunds, disputes/support, and business-seller status. Link contextually beside reservation and owner CTAs.
- **Acceptance:** each material promise links to approved policy text with owner, effective date, locale parity, and app/web consistency.
- **Complexity:** L, product/legal/content. **Release blocking: yes if these policies are not already approved elsewhere.**

#### PERF-001 — Mobile LCP is not yet launch-grade under throttling

- **Affected:** homepage; throttled mobile Chromium.
- **Problem:** local mobile Lighthouse measured 5.68 s LCP and 77 performance, despite CLS 0 and low TBT. Broken marketplace imagery, render-blocking styles, and network dependence contribute; one desktop run reached 1.21 s LCP.
- **Impact:** paid-mobile bounce and Core Web Vitals risk. Lab variance means field data—not the best run—must decide release confidence.
- **Fix:** clean the dataset, repeat cold-cache runs from the production CDN, instrument web-vitals, compress/resize below-fold imagery, trim unused client code/CSS, and enforce a hero/LCP asset budget.
- **Acceptance:** p75 field/mobile LCP ≤2.5 s, CLS ≤0.1, INP ≤200 ms; five cold mobile lab runs have median LCP ≤2.5 s with no failed requests.
- **Complexity:** M–L. **Release blocking: conditional; yes for paid launch without passing production measurements.**

### Medium

#### PRIV-001 — Google Maps loads before an explicit map choice

- **Affected:** listing detail handover/location section; all browsers.
- **Problem:** when the configured key and city exist, the Google Maps iframe is embedded directly. This shares request metadata with Google before a user explicitly activates the map.
- **Impact:** privacy/ePrivacy compliance depends on controller configuration, legal basis, consent regime, and disclosed processors. This is a legal-review issue, not a reason to remove Maps.
- **Fix:** **keep Google Maps**; legal/privacy review the embed and privacy notice. If consent is required, render the existing location fallback first and load the iframe after “Load Google Maps.” Restrict the API key by API and production referrer and set quota alerts.
- **Acceptance:** approved documented basis; privacy copy names the integration; consent behavior matches policy; key restrictions verified; denied/blocked Maps leaves a useful location summary.
- **Complexity:** M. **Release blocking: depends on qualified legal review.**

#### ANA-001 — Funnel analytics is incomplete

- **Affected:** homepage, feed, listing, modal, `/go`, QR, referral.
- **Problem:** client events cover hero search, filters, listing click, app bridge open, smart-link/store clicks. There is no reliable server-side `/go` redirect outcome, failed-app-open/retry, QR scan correlation, booking-intent completion, owner-listing completion, or native continuation event.
- **Impact:** install counts can be optimized while qualified booking/listing intent and attribution loss remain invisible.
- **Fix:** define an event contract with pseudonymous journey ID, placement, locale, safe target type/id, campaign, OS, redirect outcome, native continuation, booking intent, and owner intent. Do not log full sensitive URLs.
- **Acceptance:** deduplicated test journey is traceable from landing to native continuation; failures have explicit outcomes; consent/cookieless rules are documented; dashboards reconcile store/native totals.
- **Complexity:** L, web + native + analytics. **Release blocking: no for limited launch; yes for paid acquisition measurement.**

#### SEC-001 — CSP still allows inline script and style

- **Affected:** global response headers.
- **Problem:** production CSP uses `script-src 'unsafe-inline'` and `style-src 'unsafe-inline'`. Other headers are present and `X-Powered-By` is now removed, but this weakens CSP’s XSS containment.
- **Impact:** an injection defect has a wider execution path.
- **Fix:** migrate bridge bootstrap/inline styles toward hashes or nonces compatible with this Next.js deployment; evaluate Next SRI support; report-only first, then enforce. Do not break ISR/static delivery to achieve a cosmetic policy.
- **Acceptance:** enforced CSP contains no script `unsafe-inline`; normal navigation, hydration, structured data, analytics, Maps, error pages, and app bridge produce no CSP violations.
- **Complexity:** L. **Release blocking: no; security hardening before scale.**

#### SEO-002 — Removed-listing lifecycle cannot distinguish 404 from 410

- **Affected:** `/skelbimai/[id]` and shared listing URLs.
- **Problem:** unavailable IDs render a helpful 404, but the web/API contract does not expose a permanent-deletion state suitable for 410. Paused, expired, deleted, and malformed links therefore cannot have distinct crawler/user behavior.
- **Impact:** stale URLs may persist; owners/renters receive less precise recovery guidance.
- **Fix:** define backend states. Keep paused/temporarily unavailable pages with status/alternatives; return 410 for permanently removed listings after retention rules; 404 for unknown IDs.
- **Acceptance:** automated tests cover active, paused, expired, deleted, malformed, and backend-error states with correct HTTP, robots, copy, and next action.
- **Complexity:** M, API contract. **Release blocking: no.**

#### UX-001 — Offline and API-failure messaging is not connection-specific

- **Affected:** dynamic feed/category/listing templates.
- **Problem:** error boundaries and empty states exist, but there is no offline-specific path or retry guidance and no service-worker offline shell. A failed API can resemble empty inventory.
- **Impact:** users may conclude there are no items instead of retrying connectivity.
- **Fix:** distinguish network/offline, timeout, empty, and server errors; retain current query; show retry and browse-categories actions; avoid claiming “no results” on transport failure.
- **Acceptance:** forced offline/timeout/5xx tests announce the state, preserve inputs, retry successfully, and never emit indexable empty content.
- **Complexity:** M. **Release blocking: no.**

#### CONTENT-001 — Several LT/EN lines still require marketing/legal approval

- **Affected:** home, how-it-works, owner/renter acquisition, FAQ, account deletion.
- **Problem:** source comments explicitly mark multiple strings `DRAFT`, including trust and owner-side fee/payout/damage wording.
- **Impact:** polished language can still diverge from actual product rules; post-launch copy churn affects trust and metadata.
- **Fix:** run bilingual native-language, product, support, and legal sign-off; freeze approved strings and metadata before screenshots/store campaigns.
- **Acceptance:** no unresolved `DRAFT` markers in user-facing dictionaries; LT and EN claims are semantically equivalent, natural, and policy-backed.
- **Complexity:** M. **Release blocking: yes for claims; otherwise should-fix.**

#### ASSET-001 — App imagery is not fully locale/current-state neutral

- **Affected:** desktop/tablet homepage hero and acquisition illustrations.
- **Problem:** the phone capture uses app-specific content that may not match the English page and can age independently of the current app/store state.
- **Impact:** English visitors see weaker localization; stale ratings/status/UI erode credibility.
- **Fix:** provide approved LT and EN captures or a locale-neutral product montage; remove transient status-bar/test content; version assets with the app release.
- **Acceptance:** locale review at all breakpoints; image contents match shipping app and contain no old status/rating/test data.
- **Complexity:** M, design/content. **Release blocking: no.**

#### PERF-002 — Avoidable client and image bytes remain

- **Affected:** homepage/feed; mobile.
- **Problem:** Lighthouse estimates about 75 KiB unused JavaScript, 11 KiB unused CSS, and 41 KiB image savings. Responsive `sizes` and the priority pattern were corrected, but card sources still over-deliver at some widths.
- **Impact:** slower lower-end devices and metered connections.
- **Fix:** profile bundle ownership, keep server-only sections out of client boundaries, split feed/filter UI where practical, generate closer image widths/quality, and recompress the section pattern.
- **Acceptance:** bundle diff documents ownership; no regression to interaction behavior; median mobile transfer and LCP meet budgets.
- **Complexity:** M. **Release blocking: no after PERF-001 passes.**

#### COMPAT-001 — Required physical-engine/device matrix remains external

- **Affected:** iOS Safari, Android Chrome, Safari/Firefox/Edge desktop, social in-app browsers, screen readers, 200% zoom/text scaling.
- **Problem:** Chromium automation and source review cannot substantiate all required platforms. Next.js targets modern Chrome/Edge/Firefox/Safari, but app handoff and sticky/safe-area behavior are integration-sensitive.
- **Impact:** launch-only defects can survive despite strong automated scores.
- **Fix/acceptance:** execute and sign the matrix in Release Plan; include VoiceOver/TalkBack, keyboard, reduced motion, 200% zoom, landscape, private mode, blocked analytics/cookies/Maps, and slow/offline networks.
- **Complexity:** M QA. **Release blocking: yes for app-handoff rows; otherwise conditional.**

## Release plan

### Must fix before release

1. **Freeze identity:** confirm iOS bundle ID, Android package ID, Team ID, Play signing and upload certificates, OneLink template, and claimed paths.
2. **Repair association artifacts:** publish exact AASA and valid Asset Links; verify from public internet with no redirects.
3. **Clean production data:** remove synthetic listings, repair/purge image objects, validate active/paused/deleted inventory.
4. **Complete claims review:** product/legal/support approve LT and EN fees, deposits, refunds, cancellations, payouts, disputes, protection scope, private/business-seller wording, and Maps/privacy disclosure.
5. **Deploy the candidate atomically:** application, rewrites, headers, well-known files, sitemap/robots, environment variables, and CDN invalidation in one release.
6. **Run signed-device handoff matrix:** installed/not installed/update required; iOS/Android; Safari/Chrome; Notes/Messages/Gmail/social webviews; QR; listing/referral/campaign/category/city targets; logged in/out; clean install; interrupted store; retry/back/refresh.
7. **Run production smoke/CWV gate:** all sitemap URLs, representative dynamic states, checkout-intent bridge, error monitoring, cold mobile Lighthouse, and field web-vitals.
